# Quick Design Spec: Movement (Space Runner)

**Type**: Rework of existing system
**Scope**: Read input, compute auto-run + lateral-dodge velocity, drive the player's Rapier body, handle ground/air state, handle jump, handle slide (timed state + speed boost). Trigger animation state transitions on the player's Animation Controller. Zero camera math, zero combat, zero networking.
**Date**: 2026-04-09 (Space Runner pivot rework)
**Status**: Specified — ready for implementation
**Predecessor**: `design/archive/mml-dungeon-crawler/movement-2026-04-09.md` (MML WASD camera-relative free-move design — superseded by this spec)
**Estimated Implementation**: ~1.5 hours (smaller than the MML version — no camera math, no aim accumulator, no face-direction logic)

---

## Overview

Movement is the bridge between input (A/D lateral, Space jump, Shift slide) and the player's physics body in Space Runner. It runs in the fixed-timestep loop (`onBeforeStep`), applies a **constant forward velocity** every tick (auto-run), lerps the lateral velocity toward the requested dodge direction, handles edge-triggered jump, handles timed slide, and tells the Animation Controller which locomotion state to play.

Unlike the MML version this replaces, Movement in Space Runner:
- **Ignores the camera** — forward is a fixed world axis (-Z), not camera-relative
- **Never accumulates mouse aim** — there is no aim, no free-look, no combat lock-on
- **Never rotates the player body** — the body is angular-locked on all three axes; the mesh always faces forward
- **Has no "W held" intent** — forward speed is always on; the player runs automatically
- **Adds a slide state** — a timed behavioural state with a forward speed boost (duration and boost come from `movement.json`)

## Core Rules

### 1. Constant forward velocity (the "auto-run")

Forward is world-space -Z. Every tick, the X-unused Z component of linear velocity is set to `-config.forwardSpeed`. If the slide state is active, an additional `-config.slideSpeedBoost` is applied for the duration of the slide.

```ts
const desiredForwardZ = -config.forwardSpeed + (isSliding ? -config.slideSpeedBoost : 0);
```

No acceleration smoothing on the forward axis — the player is always at full run speed. Smoothing would introduce lag on planet-transition speed bumps later.

### 2. Lateral dodge with acceleration smoothing

Left/right dodge reads the `moveLeft` and `moveRight` actions. Unlike forward, lateral is smoothed with a rate-based exponential lerp so the dodge feels weighted rather than twitchy.

```ts
const lateralAxis = (isActionDown('moveRight') ? 1 : 0) - (isActionDown('moveLeft') ? 1 : 0);
const desiredLateralX = lateralAxis * config.lateralSpeed;

const lerpRate = isGrounded ? config.lateralAcceleration : config.lateralAcceleration * config.airControl;
const alpha = 1 - Math.exp(-lerpRate * fixedDt);

const linvel = body.linvel();
const newLateralX = linvel.x + (desiredLateralX - linvel.x) * alpha;

body.setLinvel({ x: newLateralX, y: linvel.y, z: desiredForwardZ }, true);
```

Only X (lateral) is smoothed. Z (forward) is set directly each tick. Y (vertical) is left alone so gravity and jump impulse work naturally.

### 3. Grounded check + jump (edge-triggered)

Unchanged from the MML version:

- Each tick, raycast from capsule bottom (`pos.y - 0.9`) straight down by `groundCheckDistance` (default 0.15 m).
- Jump is edge-triggered via `onActionPressed('jump')`. A held Space bar does not retrigger.
- A "just-jumped-in-ground-window" lockout prevents double-jumps in the 1-2 ticks after liftoff while the ground raycast still reports grounded.
- The lockout clears when `linvel.y <= 0` (peak of arc or falling).
- On jump: `body.setLinvel({ x, y: jumpVelocity, z }, true)` and `anim.play('jump')`.

### 4. Slide (timed state + forward boost)

Slide is a **new** Space Runner mechanic:

- Edge-triggered via `onActionPressed('slide')`.
- Only triggers if: grounded AND not already sliding AND player is alive.
- Starts a countdown timer: `slideTimeRemaining = config.slideDuration`.
- While `slideTimeRemaining > 0`:
  - Each tick, decrement by `fixedDt`.
  - Apply `-config.slideSpeedBoost` to the Z component of desired forward velocity (additional forward push).
  - Animation Controller plays `sprint` (which maps to `RunFast` in player.json) — visual placeholder until a proper slide clip is added.
- When `slideTimeRemaining <= 0`:
  - Clear the slide state.
  - Animation Controller returns to `run`.

The slide does NOT modify the capsule collider height in v1 (would require destroying and recreating the collider mid-frame). Future: add a second collider and swap between them. For v1, slide is a behavioural state without collision-shape change. Obstacles that require literal ducking are not in the v1 hazard set.

### 5. Disabled / downed / dead states

If `player.getState() !== 'alive'` OR `enabled === false`:
- Freeze velocity: `body.setLinvel({ x: 0, y: linvel.y, z: 0 }, true)`
- Skip input reads, skip slide tick, skip animation transitions.
- (Gravity still pulls the player down — only X/Z are zeroed.)

### 6. Animation state transitions

Called after velocity update each tick. Skip if current anim state is a one-shot (`jump`, `hit`, `death`).

Priority (first match wins):
1. If `isSliding` → play `sprint` (if not already)
2. Else → play `run` (if not already)

`idle` is never reached during active gameplay because forward velocity is always on. `idle` only shows when the player is downed/dead/disabled (handled by the state-gate in rule 5).

## Public API Surface (LOCKED contract)

```ts
export interface MovementConfig {
  forwardSpeed: number;         // constant forward m/s (world -Z)
  lateralSpeed: number;         // max lateral m/s (world ±X)
  lateralAcceleration: number;  // 1/sec rate constant for lateral lerp
  jumpVelocity: number;         // upward m/s impulse on jump
  airControl: number;           // 0..1 multiplier on lateral lerp while airborne
  slideDuration: number;        // seconds a slide lasts
  slideSpeedBoost: number;      // extra forward m/s while sliding
  groundCheckDistance: number;  // downward raycast length for grounded check
}

export interface MovementController {
  update(fixedDt: number): void;
  isSliding(): boolean;
  setEnabled(enabled: boolean): void;
  isEnabled(): boolean;
  dispose(): void;
}

export function createMovementController(
  engine: EngineHandle,
  player: Player,
  input: InputManager,
  config: MovementConfig,
): MovementController;
```

**Removed from MML version:**
- `getAimDirection()` — no aim in Space Runner
- Mouse sensitivity / pitch config fields
- Camera-relative forward/right derivation
- Yaw-follow (face movement direction)

**New in Space Runner version:**
- `isSliding()` public query — needed by HUD to show slide cooldown state and by future Super Suit Combat to prevent attack during slide
- `slide` action consumed from Input Manager

## Data File: `assets/data/movement.json`

```json
{
  "forwardSpeed": 12.0,
  "lateralSpeed": 7.0,
  "lateralAcceleration": 25.0,
  "jumpVelocity": 6.5,
  "airControl": 0.6,
  "slideDuration": 0.6,
  "slideSpeedBoost": 4.0,
  "groundCheckDistance": 0.15
}
```

**Tuning rationale:**
- `forwardSpeed: 12.0` — faster than MML walk (5) because the player should feel like they're running for their life
- `lateralSpeed: 7.0` — slightly more than half of forward so lateral dodges feel responsive but aren't cheap
- `lateralAcceleration: 25.0` — snappier than MML (15) for tight dodge control
- `jumpVelocity: 6.5` — slightly higher than MML (6.0) because the player is moving faster horizontally and jumps need to cover more horizontal ground
- `airControl: 0.6` — more mid-air control than MML (0.4) because lateral dodge mid-jump is a core verb
- `slideDuration: 0.6` — short enough to feel responsive, long enough to clear a typical obstacle chunk
- `slideSpeedBoost: 4.0` — ~33% extra forward speed; enough to feel like a commitment

## Dependencies

- **Player System** (reads body, anim, state)
- **Input Manager** (consumes `moveLeft`, `moveRight`, `jump`, `slide` actions)
- **Engine Bootstrap** (onBeforeStep registration, world for ground raycast)

Does **not** depend on: Camera Rig, Audio, Animation Controller directly (goes through `player.anim`).

## Input Action Requirements

New action needed: `slide`. Must be added to `src/engine/input.ts` `ActionName` union, `DEFAULT_BINDINGS` (suggested: `ShiftLeft`), and `ALL_ACTIONS` array.

Existing actions reused: `moveLeft`, `moveRight`, `jump`.
Unused but still bound (cleanup deferred): `moveForward`, `moveBack`, `shoot`, `chargeStart`, `chargeRelease`, `subWeapon`, `aimX`, `aimY`.

## Implementation Notes

- Zero-allocation pattern (module-scope Vector3 temps) — still required.
- `ALL angular DOF are locked` at spawn in `main.ts` — `setEnabledRotations(false, false, false, true)`. The body cannot tumble (X/Z) OR yaw (Y). Camera rig no longer needs to compensate for yaw.
- Clip name `slide` is intentionally NOT added to `player.json.clipMap` — instead, Movement plays the existing `sprint` clip (maps to `RunFast`) as a placeholder during slide state. A proper slide clip can be added later without changing Movement code if we update `player.json` to add a `slide` clipMap entry.
- `player.json` spawn Y remains `2` so the capsule doesn't intersect the ground on spawn.

## Acceptance Criteria

1. **Auto-run:** Player constantly moves in -Z direction at `forwardSpeed` without any input.
2. **Lateral dodge:** A moves left (-X), D moves right (+X), both smoothed with `lateralAcceleration`. Releasing returns lateral velocity to 0 smoothly.
3. **Jump:** Space applies `jumpVelocity` upward impulse once per press. No double-jump. Works while auto-running.
4. **Slide:** Shift starts a `slideDuration`-second window during which forward velocity gains `slideSpeedBoost`. Animation plays `sprint` during the window, returns to `run` after. Cannot retrigger until slide ends.
5. **Airborne lateral:** Lateral control during jump is scaled by `airControl`.
6. **State gate:** When player state is not `alive`, horizontal velocity is zeroed every tick.
7. **Zero allocations:** No `new Vector3()` inside `update()`.
8. **Build clean:** `tsc -p tsconfig.client.json --noEmit` returns 0 errors.
