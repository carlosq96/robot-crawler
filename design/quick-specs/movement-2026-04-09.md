# Quick Design Spec: Movement

**Type**: New Small System
**Scope**: Read input, compute camera-relative movement direction, drive the player's Rapier body velocity, handle ground/air state and jumping, expose aim direction for lock-on. Triggers animation state transitions on the player's Animation Controller. Has zero combat or networking knowledge.
**Date**: 2026-04-09
**Estimated Implementation**: ~2 hours

---

## Overview

Movement is the bridge between input (WASD, mouse) and the player's physics body. It runs in the fixed-timestep loop (`onBeforeStep`), reads the current input state, calculates a desired velocity using camera-relative directions, applies it to the player body, and tells the Animation Controller which locomotion state to play. It also accumulates the player's aim direction (mouse-driven) so that Buster Combat's lock-on can bias toward what the player is looking at.

Movement is intentionally **small**: it does not own the player body (Player System does), it does not own input (Input Manager does), and it does not shoot anything (Buster Combat does). It is the verb "move" applied to the noun "player," nothing more.

---

## Core Rules

### 1. Camera-relative movement

WASD moves the player relative to the camera's forward direction, not world axes. This means W always moves "into the screen" no matter how the camera is oriented — the modern third-person feel.

Computation each tick:
1. Get camera forward vector, project onto the horizontal plane (zero out Y), normalize
2. Get camera right vector (perpendicular to forward, also horizontal), normalize
3. `desiredDir = forward * (W - S) + right * (D - A)`
4. Normalize `desiredDir` if magnitude > 0 (prevents diagonal speed boost)
5. `desiredHorizontalVelocity = desiredDir * walkSpeed`

### 2. Velocity-based control with acceleration smoothing

Pure velocity-set is snappy but feels twitchy. Pure force-based is mushy. Compromise: lerp current horizontal velocity toward `desiredHorizontalVelocity` using `acceleration` (when input present) or `deceleration` (when no input):

```ts
const currentVel = body.linvel();
const horizontalCurrent = new Vector3(currentVel.x, 0, currentVel.z);
const lerpRate = isInputActive ? config.acceleration : config.deceleration;
const newHorizontal = horizontalCurrent.lerp(desiredHorizontalVelocity, lerpRate * dt);
body.setLinvel({ x: newHorizontal.x, y: currentVel.y, z: newHorizontal.z }, true);
```

Note: **only the horizontal velocity is overwritten**. The vertical velocity (Y) is left alone so gravity and jumps work naturally.

### 3. Grounded check + jump

Each tick, raycast straight down from the capsule bottom by `groundCheckDistance` (default: 0.15 m). If the ray hits a collider, the player is grounded.

```ts
const ray = new RAPIER.Ray(
  { x: pos.x, y: pos.y - capsuleHalfHeight, z: pos.z },
  { x: 0, y: -1, z: 0 }
);
const hit = world.castRay(ray, config.groundCheckDistance, true);
const isGrounded = hit !== null;
```

If `input.isActionDown('jump')` AND `isGrounded`:
- Set body linvel Y to `config.jumpVelocity` (instant upward push)
- Trigger jump animation (deferred — jump animation is post-vertical-slice)

While in the air, horizontal control is multiplied by `airControl` (0..1) so the player can adjust direction mid-jump but not freely accelerate.

### 4. Aim direction accumulation

Each tick, read `input.consumeMouseDelta()` and add it to an internal `yawAccumulator` and `pitchAccumulator` (both clamped). The aim direction is computed from these as a unit vector in world space.

```ts
const { dx, dy } = input.consumeMouseDelta();
yawAccumulator -= dx * config.yawSensitivity;       // negate to match screen X
pitchAccumulator = clamp(pitchAccumulator - dy * config.pitchSensitivity, -PI/3, PI/3);

const aimDir = new Vector3(
  Math.sin(yawAccumulator) * Math.cos(pitchAccumulator),
  Math.sin(pitchAccumulator),
  Math.cos(yawAccumulator) * Math.cos(pitchAccumulator)
);
```

This is **purely a query** for Buster Combat — Movement does not shoot. Buster Combat calls `movement.getAimDirection()` and uses the result to bias which enemy gets locked on.

### 5. Animation state transitions

Each tick after applying velocity:
- If `|horizontalVelocity| > 0.1` and player is grounded → ensure `walk` is playing
- If `|horizontalVelocity| < 0.1` and player is grounded → ensure `idle` is playing
- Don't redundantly call `play()` if already in the target state (Animation Controller handles this but it's polite to skip)

Run vs walk distinction is deferred to a stretch upgrade or future Movement v2.

### 6. Disable when player is downed/dead

```ts
if (player.getState() !== 'alive') {
  body.setLinvel({ x: 0, y: body.linvel().y, z: 0 }, true); // freeze horizontal
  return;
}
```

When the player is downed or dead, Movement zeroes horizontal velocity each tick (so they don't slide on slopes) and skips all input processing.

### 7. Hooked into Engine Bootstrap fixed step

Movement registers `engine.onBeforeStep(...)` so its update runs **before** the Rapier `world.step()`. This way the velocity it sets is applied during the same physics step, not a frame later.

### 8. Disposal

`movement.dispose()`:
- Unregisters the `onBeforeStep` callback
- Idempotent

## Public API Surface

```ts
// src/gameplay/movement.ts
import * as THREE from 'three';
import type { EngineHandle } from '../engine/bootstrap.js';
import type { InputManager } from '../engine/input.js';
import type { Player } from './player.js';

export interface MovementConfig {
  walkSpeed: number;            // m/s on horizontal plane
  acceleration: number;         // velocity lerp rate when input active (1/sec)
  deceleration: number;         // velocity lerp rate when no input (1/sec)
  jumpVelocity: number;         // upward m/s on jump
  airControl: number;           // 0..1 — horizontal control multiplier in air
  groundCheckDistance: number;  // m below capsule bottom
  yawSensitivity: number;       // radians per pixel of mouse delta X
  pitchSensitivity: number;     // radians per pixel of mouse delta Y
  pitchClampDegrees: number;    // max up/down look angle
}

export interface MovementController {
  update(fixedDt: number): void;
  getAimDirection(): THREE.Vector3;  // normalized world-space direction
  setEnabled(enabled: boolean): void;
  isEnabled(): boolean;
  dispose(): void;
}

export function createMovementController(
  engine: EngineHandle,
  player: Player,
  input: InputManager,
  config: MovementConfig
): MovementController;
```

Consumer pattern:

```ts
// In main.ts after player + input are set up
const movement = createMovementController(engine, player, input, configFromJSON);
// No need to wire onBeforeStep — the factory does it internally.
```

## Tuning Knobs

All values live in `assets/data/movement.json` per the `gameplay-code` rule.

| Knob | Default | Range | Category | Rationale |
|---|---|---|---|---|
| `walkSpeed` | `5.0` | 2 — 10 | feel | m/s — brisk walking pace; matches MML feel |
| `acceleration` | `15.0` | 5 — 50 | feel | High = snappy, low = sluggish |
| `deceleration` | `20.0` | 5 — 50 | feel | Slightly higher than acceleration so player stops cleanly |
| `jumpVelocity` | `6.0` | 3 — 12 | feel | m/s upward; about 1.8 m of jump height with 9.81 gravity |
| `airControl` | `0.4` | 0 — 1 | feel | Reduced air control prevents floaty corrections |
| `groundCheckDistance` | `0.15` | 0.05 — 0.5 | tech | Slightly larger than skin width to avoid false-airborne |
| `yawSensitivity` | `0.003` | 0.001 — 0.01 | feel | Radians per pixel; tunable per player later |
| `pitchSensitivity` | `0.003` | 0.001 — 0.01 | feel | Same as yaw for consistent feel |
| `pitchClampDegrees` | `60` | 30 — 89 | feel | Prevents over-rotation; below 90 to avoid camera flip |

## Acceptance Criteria

### Vertical Slice 1 (player walks around)
- [ ] `createMovementController(...)` returns a working controller
- [ ] Pressing W moves the player in the camera's forward direction
- [ ] Pressing A/D strafes correctly relative to camera
- [ ] Pressing W+D moves diagonally at the same speed as W or D alone (no diagonal speed boost)
- [ ] Releasing all movement keys causes the player to decelerate to a stop within ~0.2 s
- [ ] Animation Controller plays `walk` while moving and `idle` while stopped
- [ ] Camera Rig follows the player smoothly (verified by visual inspection)

### Jump
- [ ] Pressing Space while grounded launches the player upward
- [ ] Pressing Space while in the air does nothing (no double-jump)
- [ ] Player lands and is detected as grounded again
- [ ] Horizontal control while airborne is reduced (per `airControl`)

### Aim
- [ ] Moving the mouse left/right rotates the aim direction yaw
- [ ] Moving the mouse up/down rotates the aim direction pitch
- [ ] Pitch is clamped to `±pitchClampDegrees`
- [ ] `getAimDirection()` returns a normalized Vector3

### Disable behavior
- [ ] When player state becomes `downed`, horizontal velocity is zeroed each tick
- [ ] `setEnabled(false)` halts all input processing
- [ ] Re-enabling resumes normal behavior

### Architectural
- [ ] Movement has NO imports from `src/combat/`, `src/networking/`, `src/ui/`
- [ ] Movement does NOT call `play('shoot')` or any combat-related animation
- [ ] All values from `movement.json`, no hardcoded numbers
- [ ] Update runs in `onBeforeStep`, not `onAfterStep` or `onBeforeRender`

## Out of scope (deferred)

- **Sprint / run speed** — stretch; activated by Shift, swaps to `run` animation
- **Crouch** — not in concept doc, not designed
- **Dash / dodge roll** — not in concept doc
- **Wall slide / climb** — not in concept doc
- **Shooting projectiles** — Buster Combat (uses `getAimDirection()`)
- **Lock-on highlight** — Buster Combat (uses `getAimDirection()` for bias)
- **Run animation triggering** — defer until Sprint is added
